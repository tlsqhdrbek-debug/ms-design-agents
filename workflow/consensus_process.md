# 합의 도출 프로세스

페르소나 간 의견 충돌을 어떻게 해결하고 최종 설계로 수렴시키는지 정의.

---

## 1. 충돌 유형

### 1.1 Security Veto
- 가장 빈번. Security가 망분리/개인정보 규정 위반을 발견
- 해결: 위반을 제거하는 방향으로 Architect 또는 Developer가 재작업

### 1.2 Architect vs Developer
- Architect의 배치를 Developer가 "기술적으로 구현 불가"라고 판단
- 해결: Developer가 구체적 불가 사유를 제시 → Architect 재검토

### 1.3 사용자 요구 vs 제약
- 사용자 요구가 본질적으로 망분리 규정과 충돌
- 해결: Orchestrator가 사용자에게 에스컬레이션, 요구 축소 또는 대안 협상

### 1.4 두 가지 합리적 설계의 트레이드오프
- 양쪽 모두 규정 준수, 어느 쪽을 택할지 가치 판단 필요
- 해결: Orchestrator가 사용자에게 옵션 제시

---

## 2. 재설계 루프 (최대 3회)

```
[Try 1] Architect/Developer 재설계 → Security 검토
  ├─ 통과: Documentation으로 진행
  └─ Veto: [Try 2]

[Try 2] 동일 또는 다른 페르소나 재설계 → Security 재검토
  ├─ 통과: 진행
  └─ Veto: [Try 3]

[Try 3] 재설계 → Security 재검토
  ├─ 통과: 진행
  └─ Veto: 사용자 에스컬레이션 (Orchestrator)
```

각 시도마다 Orchestrator는 시도 횟수를 명시:

```
[Orchestrator] 재설계 시도 2/3. Architect에게 외부망 컴포넌트에서 사번을 제거하고
부서 코드로 치환하는 방안을 요청합니다.
```

---

## 3. 합의 명시

충돌이 해결되어 최종 설계로 진행할 때 Orchestrator는 **합의 요약**을 명시한다.

```
[Orchestrator] 합의 완료.

최종 결정:
- 패턴: C (외부망 → 내부망 연계)
- 외부망: 법령 API 폴링, 텍스트 추출, 게이트웨이 전송
- 내부망: SharePoint 적재, 부서별 Teams 알림
- 보안 통제: 마스킹 없음 (외부 공개 데이터), 게이트웨이 승인 필요, 감사 로그 활성화

Documentation에게: 위 결정을 설계서로 작성 부탁드립니다.
```

이 요약이 Documentation의 입력이 된다.

---

## 4. 사용자 에스컬레이션 형식

3회 재설계 후에도 미해결이거나 본질적 충돌일 때:

```
[Orchestrator] 추가 결정이 필요합니다.

원 요구사항: "법령 변경 시 담당자에게 개별 메일 발송"

충돌: 외부망에서 담당자 이메일을 처리하면 개인정보 위반.

검토한 대안:

A. 개별 메일 → 부서 메일 변경
   - 외부망에서 부서 코드만 사용. 내부망에서 부서 메일링 리스트로 발송.
   - 개인 식별 알림 불가.

B. 알림 발송을 모두 내부망으로 이동
   - 외부망에서는 법령 텍스트만 추출. 내부망에서 담당자 매핑 후 발송.
   - 외부망→내부망 연계 구성 필요 (게이트웨이 승인 2~3주 소요).

C. 요구사항 축소
   - 알림 대신 SharePoint 게시판 방식으로 변경.

어느 방향으로 진행할까요?
```

사용자 응답 후 해당 옵션으로 재설계 진행.

---

## 5. 합의가 실패해도 진행하는 경우

없다. **모든 페르소나의 합의 없이는 Documentation으로 진행하지 않는다.**

특히 Security가 통과시키지 않은 설계는 절대 문서화하지 않는다. 사용자 압박이 있더라도 Orchestrator는 본 원칙을 유지해야 한다.
