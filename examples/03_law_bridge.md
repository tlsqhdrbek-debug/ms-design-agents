# 예시 3: 외부망 → 내부망 연계 — 법령 변경을 내부 직원에게 알림

## 사용자 요청 (가상)
> 법제처 API에서 우리 업종 관련 법령 변경사항을 가져와서, 컴플라이언스 담당 직원들에게 매일 아침 Teams로 알려주고 싶어요.

## 페르소나 협업 흐름

### [Orchestrator] 요구사항 분해
- 데이터 소스: 법제처 API (외부)
- 결과 수신자: 내부 직원 (컴플라이언스 담당)
- 알림 방식: Teams (내부망)
- 개인정보 포함 여부: 담당자 이름 표시 여부 확인 필요

→ 사용자 확인 후: "이름은 굳이 안 표시해도 됩니다. 채널 알림으로 OK."

### [Architect] 결정 트리
- Q1 (개인정보): 아니오 (담당자 식별 없음)
- Q3 (외부 의존): 예
- Q4 (수신자 = 내부 직원): 예
- Q5 (개인정보 알림 필요): 아니오
- **결정: 패턴 C — 외부망 + 내부망 연계 (단방향)**

컴포넌트:
| 컴포넌트 | 위치 | 기술 |
|---------|------|------|
| 일정 트리거 | 외부망 | Power Automate (Schedule) |
| 법제처 API 호출 | 외부망 | Power Automate (HTTP) |
| 텍스트 정제 | 외부망 | Power Automate (Compose) |
| **단방향 게이트웨이 전송** | 외부망 → 내부망 | 회사 승인 게이트웨이 (텍스트만) |
| SharePoint 적재 | 내부망 | Power Automate (SharePoint) |
| Teams 채널 알림 | 내부망 | Power Automate (Teams) |

### [Developer] 플로우 명세 (2개 플로우)

#### 외부망 플로우: `LawWatch_External`
| 순번 | 단계 | 액션 | 커넥터 |
|-----|------|------|--------|
| 1 | 트리거 | Recurrence | Schedule (매일 07:00 KST) |
| 2 | API 호출 | HTTP | HTTP |
| 3 | 파싱 | Parse JSON | Data Operations |
| 4 | 텍스트 정제 | Compose | Built-in |
| 5 | 게이트웨이 전송 | 회사 게이트웨이 커넥터 또는 메일 | 회사 규정 방식 |

#### 내부망 플로우: `LawWatch_Internal`
| 순번 | 단계 | 액션 | 커넥터 |
|-----|------|------|--------|
| 1 | 트리거 | When item created | SharePoint (게이트웨이가 적재한 리스트) |
| 2 | 또는 | When email arrives | Outlook (게이트웨이가 보낸 메일) |
| 3 | 데이터 가공 | Compose | Built-in |
| 4 | SharePoint 적재 | Create item | SharePoint (내부망 리스트) |
| 5 | Teams 알림 | Post message in channel | Teams |

### [Security] 체크리스트
- A1 내부망 플로우에 외부 호출 없음: ✅
- A3 회사 승인 게이트웨이 사용: ⚠️ 회사 게이트웨이 방식 확인 필요
- A4 단방향 (외부→내부만): ✅
- B1 외부망에 개인정보 없음: ✅ 법령 텍스트만
- B3 알림에 개인정보 미포함: ✅ "오늘자 법령 변경 N건" 형식
- G1 이송 데이터 명시: ✅ 법령 본문 텍스트만
- G3 이송 채널 = 게이트웨이: ⚠️ 회사 승인 방식 확인

**판정: 조건부 통과** — 회사 게이트웨이 운영 부서와 사전 협의가 전제 조건.

### [Orchestrator] 합의
- 패턴 C 외부망→내부망 연계 확정
- 사전 조건: 회사 게이트웨이 부서 승인
- 두 개의 플로우로 분리 (외부망/내부망 각 1개)
- Documentation으로 진행

---

## 학습 포인트

- **연계가 필요한 경우 반드시 두 개의 플로우로 분리** (외부망 1, 내부망 1)
- 게이트웨이 방식은 회사마다 다르므로 "회사 승인 방식"으로 표기하고 운영 부서와 협의 전제 명시
- 알림 본문에 개인 식별 없으면 개인정보 이슈 없음 → 패턴 C의 가장 흔한 형태
- 만약 사용자가 "담당자 이름까지 알림에 표시"를 원했다면 Q5=예가 되어 추가 통제 필요 (예: 담당자 매핑은 내부망에서만)
